1. Executive Summary

This document is the working SEO blueprint for the new Avana Surgical Systems website. It combines the findings from the on-page SEO audit with the India-only keyword research workbook (avana-keyword-analysis.xlsx) and translates them into the exact meta titles, meta descriptions, on-page elements, and technical fixes that should be implemented before launch.

Where the audit told us what the site is doing well and where the gaps are, the keyword research tells us which phrases real Indian users are typing into Google. This document brings the two together so every page is built around (a) a clearly identified head keyword, (b) a supporting cluster, and (c) the right intent stage.


Key insights driving these recommendations


URL strategy: Pain-led slugs win. /knee-pain captures ~165K monthly searches in India; /knee-brace captures only ~27K. The same pattern holds for back, shoulder, foot, elbow and hip. The current /solutions/{slug} structure is acceptable, but a flatter /{area}-pain pattern would convert better long-term.


Three-tier architecture: Pain hubs (awareness + commercial) → Condition pages (high-intent informational) → Audience pages (commercial). All three should funnel transactional clicks to osteokart.com.


The 5 single highest-volume head terms in India are: back pain (246K), knee pain (165K), lower back pain (110K), knee cap (90K) and foot pain (60.5K). Every meta title for these pages must contain the head term verbatim.


Title and description hygiene: 5 current page titles and 4 descriptions risk being truncated by Google. All have been rewritten in this document to land inside the 50-60 char (title) and 150-160 char (description) sweet spot.


Quick wins first: robots.txt, sitemap.xml, Open Graph tags, canonical links, and MedicalBusiness JSON-LD on the homepage are missing today. These are 30 minutes of work and unlock significant ranking + sharing value.


How to use this document


Section 2 — Meta titles and descriptions for every existing page, with character counts and reasoning.


Section 3 — Keyword-to-page map (which keyword lives on which page).


Section 4 — Page-by-page on-page SEO requirements (H1, H2s, FAQs, internal links, schema).


Section 5 — Technical SEO fixes (robots.txt, sitemap, OG tags, schema).


Section 6 — 90-day publishing roadmap so the team knows what to ship in what order.

4. On-Page SEO Requirements

Each page must satisfy a minimum on-page SEO checklist. Use this section as a build spec.


4.1 Universal page checklist

Every page on the site must have:


Exactly ONE <h1> containing the page&apos;s primary keyword.


A logical H2/H3 hierarchy. H2s should contain secondary keywords from the cluster.


Meta title (50-60 chars) and description (150-160 chars) per Section 2.


<link rel=&apos;canonical&apos; href=&apos;https://avanasurgical.com/...&apos;> pointing to the live URL.


Open Graph tags: og:title, og:description, og:image (1200×630), og:url, og:type.


Twitter Card meta: twitter:card=&apos;summary_large_image&apos;, twitter:title, twitter:description, twitter:image.


<html lang=&apos;en-IN&apos;> (currently &apos;en&apos; — change to &apos;en-IN&apos; to reinforce India targeting).


Internal link to at least 2 other Avana pages and 1 osteokart.com category page.


First 100 words of body copy must contain the primary keyword once.


Image alt text using descriptive, keyword-aware phrasing (not just &apos;image&apos; or &apos;product&apos;).


Structured data appropriate to the page type (see Section 5.4).


4.2 Pain hub page template

Each /solutions/{slug} page (knee, back, shoulder, elbow, foot, hip) should follow this structure:

Section

Content & SEO purpose

H1

Contains primary head term + &apos;Pain&apos; or &apos;Pain Solutions&apos;. Example: &apos;Knee Pain Solutions & Bracing in India&apos;.

Intro (100-150 words)

Plain-English summary. Must repeat the primary keyword in sentence 1. Mentions Avana + India + doctor-recommended. End with a CTA: &apos;Find your solution below&apos;.

H2: Common causes of {area} pain

Targets the &apos;{keyword} causes&apos; query (e.g. knee pain causes = 14K). Use a 5-7 bullet list of causes. Each bullet ~20-30 words.

H2: {keyword} treatment options

Targets the &apos;{keyword} treatment&apos; query (knee pain treatment = 22K, back pain relief = 14K). 4-6 short paragraphs.

H2: Recommended {area} support

Product strip. 4-6 product cards. Each card&apos;s H3 = exact India keyword (knee cap, knee brace, lumbar belt, etc.). Each links to osteokart.com with UTM.

H2: Conditions we treat

Lists 4-6 condition pages (slipped disc, sciatica, etc.) with anchor text matching each condition&apos;s keyword. Each is an internal link.

H2: Who is this for?

Lists 2-3 audience pages (Seniors, Athletes, Post-Surgery). Internal links.

H2: Frequently asked questions

5-7 FAQ items with FAQPage JSON-LD schema. Use long-tail questions from People-Also-Ask (e.g. &apos;What causes knee pain in old age?&apos;).

H2: Why Avana

Trust signals: years, patients served, brands distributed, surgeon endorsements.

Footer CTA

4.5 Internal linking rules

Internal linking is the single biggest SEO lever the site currently under-uses. The audit flagged that pain-area pages don&apos;t link to audience or condition pages from in-body content. Fix this:


Every pain hub must link OUT to at least 3 condition pages (use the condition keyword as anchor text).


Every pain hub must link OUT to at least 2 audience pages (e.g. /solutions/knee → Seniors and Athletes).


Every condition page must link BACK to its parent pain hub in the first paragraph (anchor = pain hub keyword).


Every audience page must link to 2-3 relevant pain hubs (e.g. Seniors → Knee, Spine, Hip).


Every page must have one prominent CTA to an osteokart.com category page (with UTM parameters for revenue attribution).


Footer links to all 6 pain hubs (currently removed at your request — recommend adding back, this is the only place crawlers find them apart from the mega-menu).


Use varied anchor text, not the same phrase every time. Mix exact-match (&apos;knee pain&apos;), phrase-match (&apos;learn about knee pain&apos;), and branded (&apos;Avana knee solutions&apos;).

A note on the URL strategy debate

The keyword research workbook makes a strong case for migrating the pain hubs from /solutions/{slug} to /{area}-pain (e.g. /knee-pain instead of /solutions/knee). The flatter URL pattern matches user search intent more directly and is easier to remember. However, this is a Phase 2 decision — the existing URL structure ranks fine if everything else in Sections 2-5 is implemented well. Migrate only if you&apos;re prepared to handle 301 redirects properly. If you do migrate, do it before the site accumulates significant backlinks; redirects bleed ~10-15% of link equity each time.
