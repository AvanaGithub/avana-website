1. Executive Summary

This document answers two practical questions about the Avana website rollout:


How do we run a static main site (HTML/CSS/vanilla JS) and a WordPress blog under the same domain, both hosted on DigitalOcean, in a way that&apos;s clean for users, search engines, and the dev team?


How do we measure SEO and analytics performance across both — without losing data when a user moves from /blog/some-post to /solutions/knee?



The short answer:


Run BOTH sites under one domain. The static site lives at the root (avanasurgical.com) and WordPress lives at /blog (avanasurgical.com/blog). They share one SSL certificate, one Google Analytics property, one Search Console property, and one set of cookies.


Use Nginx as a reverse proxy. Nginx serves the static files for everything except /blog, where it transparently forwards to WordPress running on the same droplet (or a separate one).


Install Google Tag Manager (GTM) once via a shared snippet and load Google Analytics 4 through it. Both sites emit the same client_id, so user journeys stitch together automatically.


Add WordPress to your sitemap.xml index. Search Console then sees blog pages as part of avanasurgical.com — not as a separate property.

Why a subdirectory (/blog) and not a subdomain (blog.avanasurgical.com)?
From an SEO point of view, a subdirectory inherits the domain authority of the main site. Blog content lifts the main site&apos;s rankings and vice versa. A subdomain is treated by Google as a separate property — you&apos;d be building authority twice and tracking traffic in two places. Subdirectory wins, almost without exception, for content marketing.

3. Nginx Reverse Proxy — Step by Step

This is the technical heart of the integration. Nginx is the front door for the whole domain. It decides: "Is this request for /blog? Send it to WordPress. Otherwise, serve a static HTML file."


3.1 The big picture

A request lifecycle looks like this:

#

What happens

1

User requests https://avanasurgical.com/solutions/knee — DNS points to the droplet&apos;s public IP.

2

Nginx receives the request on port 443 (HTTPS, with the Let&apos;s Encrypt certificate).

3

Nginx checks the path. /solutions/knee does NOT start with /blog → serves /var/www/avana-static/solutions/knee/index.html directly.

4

Same user clicks the "Blog" link in the header → requests https://avanasurgical.com/blog/knee-pain-causes.

5

Nginx sees /blog → reverse-proxies to PHP-FPM (Unix socket) which runs WordPress.

6

WordPress generates HTML and sends it back through Nginx to the user.

7

Browser stores cookies under avanasurgical.com (not blog.avanasurgical.com) — so ALL analytics cookies are shared across both sites automatically.

3.3 Complete Nginx configuration

This is the full /etc/nginx/sites-available/avanasurgical.conf file. Drop it in, run `sudo nginx -t` to test, then `sudo systemctl reload nginx`.

# /etc/nginx/sites-available/avanasurgical.conf
# Avana Surgical — static main site + WordPress at /blog

# Redirect www to apex
server {
    listen 80;
    listen 443 ssl http2;
    server_name www.avanasurgical.com;
    ssl_certificate     /etc/letsencrypt/live/avanasurgical.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/avanasurgical.com/privkey.pem;
    return 301 https://avanasurgical.com$request_uri;
}

# Redirect HTTP to HTTPS
server {
    listen 80;
    server_name avanasurgical.com;
    return 301 https://avanasurgical.com$request_uri;
}

# Main HTTPS server
server {
    listen 443 ssl http2;
    server_name avanasurgical.com;

    ssl_certificate     /etc/letsencrypt/live/avanasurgical.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/avanasurgical.com/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # Security headers (apply to BOTH static + WordPress)
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # ---------- WordPress at /blog ----------
    location /blog {
        root /var/www/avana-wordpress;
        index index.php index.html;

        # Make WordPress think it lives at /blog
        try_files $uri $uri/ /blog/index.php?$args;

        location ~ \.php$ {
            include snippets/fastcgi-php.conf;
            fastcgi_pass unix:/run/php/php8.3-fpm.sock;
            fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        }

        # Don&apos;t serve sensitive WordPress files
        location ~ /\.(?!well-known).* { deny all; }
        location = /blog/wp-config.php { deny all; }
        location = /blog/xmlrpc.php { deny all; }

        # Optional: restrict wp-admin and wp-login to office IPs
        # location ~ ^/blog/(wp-admin|wp-login\.php) {
        #     allow 203.0.113.42;   # office IP
        #     deny all;
        # }

        # Cache static WordPress assets aggressively
        location ~* /blog/.+\.(css|js|jpg|jpeg|png|gif|ico|webp|svg|woff2?)$ {
            expires 30d;
            access_log off;
            add_header Cache-Control "public, immutable";
        }
    }

    # ---------- Static site at everything else ----------
    root /var/www/avana-static;
    index index.html;

    # Pretty URLs: /solutions/knee → /solutions/knee/index.html
    location / {
        try_files $uri $uri/ $uri.html $uri/index.html =404;
    }

    # Cache static assets
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|webp|svg|woff2?)$ {
        expires 30d;
        access_log off;
        add_header Cache-Control "public, immutable";
    }

    # robots.txt and sitemap.xml live in the static site root
    location = /robots.txt   { access_log off; log_not_found off; }
    location = /sitemap.xml  { access_log off; log_not_found off; }

    # Real 404 page (replace the redirect-to-homepage anti-pattern)
    error_page 404 /404.html;
    location = /404.html { internal; }

    # Gzip
    gzip on;
    gzip_vary on;
    gzip_min_length 256;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript image/svg+xml;
}


3.4 WordPress configuration to live at /blog

WordPress needs to know it lives at /blog, not at the root. Two changes:


In WordPress admin → Settings → General, set both "WordPress Address (URL)" and "Site Address (URL)" to https://avanasurgical.com/blog


In wp-config.php, force these values so they can&apos;t be tampered with:

define(&apos;WP_HOME&apos;,    &apos;https://avanasurgical.com/blog&apos;);
define(&apos;WP_SITEURL&apos;, &apos;https://avanasurgical.com/blog&apos;);
define(&apos;FORCE_SSL_ADMIN&apos;, true);

// Trust the X-Forwarded-Proto header from Nginx so WordPress knows it&apos;s HTTPS
if (isset($_SERVER[&apos;HTTP_X_FORWARDED_PROTO&apos;]) && $_SERVER[&apos;HTTP_X_FORWARDED_PROTO&apos;] === &apos;https&apos;) {
    $_SERVER[&apos;HTTPS&apos;] = &apos;on&apos;;
}

Add to nginx config under the /blog location block (already in the snippet above) — these proxy headers ensure WordPress reads the original protocol and host:

proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto https;


3.5 Deployment workflow

How the static site gets onto the droplet:


Push static-site code to a GitHub repo (private).


Add a GitHub Actions workflow that, on push to main, rsyncs /dist to /var/www/avana-static on the droplet over SSH.


WordPress is updated through its own admin UI — no GitHub deploy needed.


Both can be updated independently. Pushing a blog post doesn&apos;t touch the static site, and pushing a new homepage doesn&apos;t restart WordPress.

Example GitHub Actions snippet (.github/workflows/deploy.yml):

name: Deploy static site
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: rsync to DigitalOcean
        uses: easingthemes/ssh-deploy@main
        env:
          SSH_PRIVATE_KEY: ${{ secrets.DO_SSH_KEY }}
          REMOTE_HOST: ${{ secrets.DO_HOST }}
          REMOTE_USER: deploy
          SOURCE: "./"
          TARGET: "/var/www/avana-static/"
          ARGS: "-rltgoDzvO --delete --exclude=.git --exclude=.github"




4. Shared Design, Navigation & SEO Elements

From the user&apos;s point of view, the two sites must feel like one. Same logo, same header, same footer, same color scheme, same fonts. From Google&apos;s point of view, they must announce themselves as one property through canonical tags, sitemaps, and a shared robots.txt.


4.1 Visual consistency

Element

How to keep it consistent

Header & nav

Build a custom WordPress theme (or child of an existing one) that uses the SAME HTML structure as the static site&apos;s header. Same logo file, same nav links, same mega-menu. Easiest path: copy the static header.html into header.php, replace hardcoded links with WordPress menu functions (wp_nav_menu).

Footer

Same approach — static site&apos;s footer.html becomes footer.php.

Fonts & colors

Both sites should load the same CSS. Host the master design tokens (CSS variables for colors, font stacks, spacing) in /var/www/avana-static/assets/css/tokens.css and have the WordPress theme reference it with an absolute URL: <link rel=&apos;stylesheet&apos; href=&apos;/assets/css/tokens.css&apos;>. One file = one source of truth.

Logo & favicons

Store under /assets/img/ in the static site. WordPress references them by absolute path.

Buttons / cards

Reuse the same CSS classes. If a button is .btn-primary on the static site, it&apos;s .btn-primary on the blog. WordPress posts should be wrapped in the same .container max-width as the static pages.

Mobile menu

Build it once in vanilla JS and load the same script on both. Easier than building two separate hamburger menus.


4.2 Navigation: how to link between the two

Every page on the static site has a "Blog" link in the header pointing to /blog. Every blog post has a header link back to /solutions/{area} when relevant. Make these links contextual:


Each blog post is tagged with a parent pain area (e.g. &apos;Knee&apos;). At the bottom of the post, show a CTA card: "Looking for knee pain solutions? Visit our knee pain hub."


Each /solutions/{slug} page has a section "Latest from the blog" pulling the 3 most recent posts tagged with that pain area. WordPress can expose this via its REST API at /blog/wp-json/wp/v2/posts?categories={id} — the static page fetches it client-side with vanilla JS.


Breadcrumbs on blog posts read: Home > Blog > Knee > Post title. The &apos;Home&apos; link goes to / (static site), not /blog.

This achieves what SEO calls topical authority: Google sees that your knee content lives in multiple places (solution hub, conditions, audience page, AND blog) all interlinked. Authority compounds.


4.3 Canonical URLs and the sitemap index

With two systems generating URLs, the risk is that a post or page gets indexed twice (e.g. /blog/post and /blog/post/feed). Two safeguards:


Every page on both sites has a self-referencing canonical tag. Static pages get it hardcoded in the <head>; WordPress emits it automatically via Yoast SEO or Rank Math.


Use a sitemap INDEX (not a single sitemap) at /sitemap.xml that points to BOTH the static site sitemap AND the WordPress sitemap.

Example /sitemap.xml served by Nginx from the static site directory:

<?xml version="1.0" encoding="UTF-8"?>
<sitemapindex xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">

  <sitemap>
    <loc>https://avanasurgical.com/sitemap-static.xml</loc>
    <lastmod>2026-05-12</lastmod>
  </sitemap>

  <sitemap>
    <loc>https://avanasurgical.com/blog/sitemap_index.xml</loc>
    <lastmod>2026-05-12</lastmod>
  </sitemap>

</sitemapindex>

Then create /sitemap-static.xml with the 16 static URLs (as in the previous SEO doc), and install Yoast SEO or Rank Math on WordPress — both auto-generate /blog/sitemap_index.xml every time you publish a post.

Submit only the parent index
In Google Search Console, submit only https://avanasurgical.com/sitemap.xml. Google will follow it to both child sitemaps. Don&apos;t submit the two children separately — it muddles the indexation status view.

5.2 Why Google Tag Manager (and not direct gtag.js)

Three reasons GTM is the right choice for a two-system setup:


One snippet, two sites. You add the GTM snippet to the static site&apos;s HTML once (in a shared header.html include or directly in each <head>) and add it to the WordPress theme&apos;s header.php once. From then on, all tags are added through GTM&apos;s web UI.


Marketing team independence. The team can add a new Meta Pixel event or change the GA4 measurement ID without filing a dev ticket.


Same client_id across both sites. Because GTM loads gtag.js with the same configuration on both sites, GA4 stitches together a user who lands on /blog/knee-pain-causes and then clicks through to /solutions/knee — it&apos;s recorded as ONE session by ONE user, not two.


5.3 Step-by-step GTM + GA4 installation

Follow these steps in order:


Create a Google Analytics 4 property at analytics.google.com. Use &apos;Avana Surgical&apos; as the property name and &apos;India&apos; as the country. Note down the Measurement ID (looks like G-XXXXXXXXXX).


Inside the GA4 property, create ONE data stream for https://avanasurgical.com. Do NOT create a separate stream for /blog. One domain = one stream.


Create a Google Tag Manager container at tagmanager.google.com. Name it &apos;Avana Surgical (Web)&apos;. Note down the GTM ID (looks like GTM-XXXXXXX).


Inside GTM, create a new tag of type &apos;Google Tag&apos; (the unified gtag). Set Tag ID = your GA4 Measurement ID. Set trigger = &apos;All Pages&apos;. Publish the container.


Install the GTM snippet on both sites (see code below).


Open avanasurgical.com in an incognito window. Open GA4 → Reports → Realtime. You should see yourself as 1 active user. Click around the static site, then click through to /blog. The active user count should stay at 1 (proves stitching is working).




GTM snippet — static site

Place this in the <head> of every static page (use a build-time include if possible). Replace GTM-XXXXXXX with your real container ID:

<!-- Google Tag Manager - HEAD -->
<script>(function(w,d,s,l,i){w[l]=w[l]||[];w[l].push({&apos;gtm.start&apos;:
new Date().getTime(),event:&apos;gtm.js&apos;});var f=d.getElementsByTagName(s)[0],
j=d.createElement(s),dl=l!=&apos;dataLayer&apos;?&apos;&l=&apos;+l:&apos;&apos;;j.async=true;j.src=
&apos;https://www.googletagmanager.com/gtm.js?id=&apos;+i+dl;f.parentNode.insertBefore(j,f);
})(window,document,&apos;script&apos;,&apos;dataLayer&apos;,&apos;GTM-XXXXXXX&apos;);</script>
<!-- End Google Tag Manager -->

And immediately after the opening <body>:

<!-- Google Tag Manager (noscript) -->
<noscript><iframe src="https://www.googletagmanager.com/ns.html?id=GTM-XXXXXXX"
height="0" width="0" style="display:none;visibility:hidden"></iframe></noscript>
<!-- End Google Tag Manager (noscript) -->


GTM snippet — WordPress

Easiest method: install the free plugin GTM4WP (Google Tag Manager for WordPress). Paste your GTM-XXXXXXX into its settings. Activate. Done.

Manual method: edit the active theme&apos;s header.php and add the same <script> block above just before </head>, and the <noscript> block just after <body>.

Critical: use the SAME GTM ID on both sites
Both the static site and WordPress must use the exact same GTM-XXXXXXX container. This is what stitches sessions together. If you use two different containers, you&apos;ll see double the users and half the journey on each one.


5.4 Cookie domain — the silent killer of cross-site stitching

Because both sites are under avanasurgical.com (no subdomain), GA4&apos;s _ga cookie is automatically scoped to avanasurgical.com and is sent on every request to both /solutions/* and /blog/*. Nothing extra to configure. THIS is why the subdirectory approach is so much better than a subdomain: it gives you cross-site analytics for free.

If you were on blog.avanasurgical.com instead, you&apos;d need to manually configure cross-domain tracking in GTM (linker config, conversion_linker tag, allow_linker: true) — error-prone and easy to forget.


5.5 Google Search Console — single property

Set up the Search Console property as DOMAIN verification (not URL prefix). This is one click in Cloudflare or your DNS provider — add a TXT record:


Go to search.google.com/search-console → Add property → Domain.


Enter avanasurgical.com (no https://, no www, no /blog).


Google gives you a TXT record. Paste it into your DNS provider&apos;s records.


Click Verify. Done.


Submit your sitemap.xml (the parent index) in Sitemaps → Add new sitemap.

With Domain verification you see indexation, queries, and clicks for the entire property — including the WordPress /blog/* URLs — in one dashboard. No need for separate properties.

Why Domain verification, not URL Prefix?
URL prefix verification only covers exactly the URL you specify. avanasurgical.com (URL prefix) would NOT cover https://avanasurgical.com/blog or www.avanasurgical.com. Domain verification covers all subdomains and all protocols. It&apos;s a one-time setup that saves headaches forever.


5.6 Microsoft Clarity (free heatmaps + recordings)

Clarity is genuinely free with no traffic limits and no opt-out for users (it runs alongside GA4 cleanly). It gives you:


Heatmaps showing where users click on each page.


Session recordings (anonymised) — watch real people use the site.


Rage clicks, dead clicks, JavaScript error tracking.

Install via GTM (one tag, one trigger = &apos;All Pages&apos;). Use the same Clarity Project ID on both sites — same stitching benefit as GA4.

6.2 Example: tracking clicks to osteokart.com

Walkthrough of setting this up in GTM — repeat the pattern for any other event:


In GTM, create a new Variable of type &apos;Auto-Event Variable&apos; or use the built-in &apos;Click URL&apos; variable.


Create a new Trigger: type = &apos;Click - Just Links&apos;. Fires on: Some Link Clicks. Condition: Click URL contains &apos;osteokart.com&apos;.


Create a new Tag: type = &apos;Google Analytics: GA4 Event&apos;. Event name = click_osteokart. Parameters: link_url = {{Click URL}}, source_page = {{Page Path}}.


Trigger = the trigger you just created. Save and Publish.


Test in Preview mode by clicking an osteokart link on the staging site. You should see click_osteokart fire in GA4 Realtime within 30 seconds.

Same pattern, slight variation, gives you click_phone (Click URL starts with &apos;tel:&apos;), click_whatsapp (starts with &apos;wa.me&apos; or contains &apos;whatsapp.com&apos;), blog_to_solution_click (Page Path contains &apos;/blog/&apos; AND Click URL contains &apos;/solutions/&apos;).


6.3 UTM tags on every osteokart.com link

When a user clicks from avanasurgical.com to osteokart.com, the destination site sees a new session (osteokart.com is a different domain, so cookies don&apos;t cross). To attribute revenue back to Avana you need UTM tags on every outbound link.

Pattern:

https://osteokart.com/knee-cap?utm_source=avanasurgical&utm_medium=referral&utm_campaign=knee_hub&utm_content=product_strip_card_1

Parameter

Value to use

utm_source

avanasurgical  (the site they came from)

utm_medium

referral  (free traffic from another site)

utm_campaign

Page slug. Examples: knee_hub, back_hub, frozen_shoulder_condition, knee_pain_blog_post

utm_content

Position on page. Examples: product_strip_card_1, footer_cta, inline_cta, header_button

Then in osteokart&apos;s Google Analytics, you&apos;ll see clean attribution: &apos;avanasurgical / referral&apos; showing which Avana page drove revenue. This closes the loop on SEO ROI.

click_osteokart  →  proxy for revenue (refine later by integrating osteokart&apos;s actual purchase data)


form_submit  →  direct lead


click_phone  →  direct lead


click_whatsapp  →  direct lead

7. Performance, Security & Operations


7.1 Performance — keep both sites fast

Google uses Core Web Vitals as a ranking factor. A slow blog can drag down your whole domain&apos;s SEO. Targets to aim for:

Metric

Target

How to achieve it

LCP (Largest Contentful Paint)

< 2.5s

Compress hero images to <100KB. Use WebP. Lazy-load below-fold images (already done — keep it).

INP (Interaction to Next Paint)

< 200ms

Keep JavaScript light. Defer non-critical scripts. WordPress: limit plugins (5-7 max).

CLS (Cumulative Layout Shift)

< 0.1

Always set width and height attributes on <img> tags. Reserve space for ads/embeds.

TTFB (Time to First Byte)

< 600ms

Static site is already instant. For WordPress, enable Redis object cache + a page cache plugin (W3 Total Cache or LiteSpeed Cache or WP Super Cache).



WordPress performance plugins to install:


WP Super Cache or W3 Total Cache — page caching. Cuts TTFB from 800ms to <100ms.


Redis Object Cache (with Redis installed on the droplet) — caches database queries.


ShortPixel or Smush — automatic image compression on upload.


Asset CleanUp (free) — disable plugin CSS/JS on pages that don&apos;t need them.

Optional: put DigitalOcean&apos;s CDN (Spaces with CDN) or Cloudflare (free tier) in front of the whole domain. Cloudflare alone usually halves global latency for free, and gives you DDoS protection.


7.2 Security

WordPress is the #1 hacked CMS on the internet, mostly because of outdated plugins. The static site is essentially uncrackable, but a compromised WordPress on the same droplet is bad news for both. Minimum security:


Restrict /blog/wp-admin and /blog/wp-login.php to office IP addresses (the commented section in the Nginx config above).


Enable 2FA on every WordPress admin account. Plugin: Wordfence or Two Factor Authentication.


Auto-update WordPress core, plugins, and themes. WordPress does this by default — keep it on.


Install a security plugin: Wordfence (free tier) or Sucuri Security. Both block common attack patterns.


Disable XML-RPC unless you actively need it (Nginx config above already blocks /blog/xmlrpc.php). It&apos;s a frequent attack vector.


Limit login attempts: plugin Limit Login Attempts Reloaded. Stops brute force.


Use strong unique passwords (1Password / Bitwarden) and rotate them annually.


Audit installed plugins quarterly. Remove anything not actively used. Each plugin is attack surface.


Keep your droplet&apos;s OS patched: &apos;sudo apt update && sudo apt upgrade&apos; weekly, or enable unattended-upgrades.

9. Summary & Order of Operations

If everything above feels like a lot, here&apos;s the minimum sequence to get to a working, measurable, integrated setup:


Week 1 — Infrastructure


Spin up DigitalOcean droplet (Basic 2GB, Ubuntu 24.04 LTS) using the WordPress 1-Click app.


Point DNS for avanasurgical.com → droplet IP. Add www CNAME → apex.


Run certbot to get a Let&apos;s Encrypt SSL cert.


Set up the directory structure (/var/www/avana-static, /var/www/avana-wordpress).


Apply the Nginx config from Section 3.3. Test with `nginx -t`. Reload.


Set WordPress home/site URL to https://avanasurgical.com/blog in wp-config.php.


Deploy the static site via rsync / GitHub Action.


Verify both work: https://avanasurgical.com (static) and https://avanasurgical.com/blog (WordPress).


Week 2 — SEO foundation


Apply the meta titles & descriptions from the previous SEO doc to every static page.


Install Rank Math on WordPress. Run its setup wizard.


Create the sitemap index at /sitemap.xml + the static sitemap at /sitemap-static.xml. WordPress sitemap is auto-generated by Rank Math.


Create the robots.txt from Section 4.4.


Week 2 — Analytics


Create the GA4 property and the GTM container.


Install the GTM snippet on both sites (static + WordPress via GTM4WP).


Add the GA4 Google Tag inside GTM, trigger on All Pages. Publish.


Verify in GA4 Realtime that both sites are reporting under the same property.


Configure events: click_osteokart, click_phone, click_whatsapp (Section 6.2).


Mark those events as &apos;Key Events&apos; in GA4.


Verify Domain in Google Search Console. Submit sitemap.xml.


Set up Bing Webmaster Tools (auto-imports from GSC).


Install Microsoft Clarity via GTM.


Week 3 — Polish


Build the shared header/footer in WordPress matching the static site&apos;s design (Section 4.1).


Install WordPress performance plugins (WP Super Cache, Redis Object Cache, ShortPixel).


Install WordPress security (Wordfence, 2FA, Limit Login Attempts).


Set up UpdraftPlus → DigitalOcean Spaces backups.


Enable DigitalOcean droplet weekly backups.


Build the Looker Studio dashboard.


Set up UptimeRobot for both hosts.


Week 4 onwards


Publish the first blog posts per the 90-day roadmap (knee-pain-causes, frozen-shoulder-exercises, best-lumbar-belt-india, etc.).


Review the Looker Studio dashboard every Monday.


Iterate based on what users actually do — high bounce on a page? Watch a few Clarity recordings. Low click-through on a CTA? Re-word it and re-test.

The one thing not to skip
If you implement nothing else from this document, install Google Tag Manager on both sites with the same container ID before the site goes live. Every day without tracking is a day of lost data that can&apos;t be recovered. Everything else can be added later — analytics history cannot.
